# R016: Continuous Improvement Rules

**Priority**: MUST | When a rule violation is identified, update the rule IMMEDIATELY before continuing — never defer.

## Overview

R016 closes the feedback loop between violation and prevention. When the user points out a rule violation, the agent must: (1) acknowledge, (2) identify root cause, (3) update the relevant rule, (4) commit, (5) continue the original task. The rule explicitly forbids "I'll update it later" — deferral is itself a violation.

## Workflow

1. Acknowledge violation
2. Identify root cause (which rule was weak/unclear?)
3. Update the rule (add clarity, examples, self-checks)
4. Commit the change via [[mgr-gitnerd]]
5. Continue original task following updated rules

## Defect Response Matrix

| Defect Type | Rule Update | Memory | Issue |
|-------------|:-----------:|:------:|:-----:|
| Rule violation (agent behavior) | Yes | — | — |
| CI/infra defect | — | Yes | Yes |
| Process gap | Yes | Yes | Yes |
| Repeatable system bug | — | Yes | Yes |
| Agent selection failure | — | Yes | — |

## Adaptive Harness Integration

When repeating agent failures detected:
1. Record as feedback memory
2. Run `/omcustom:adaptive-harness --learn`
3. Profile updates improve future agent selection

## Anti-Patterns

| Anti-Pattern | Why Wrong |
|-------------|-----------|
| "I'll update it later" | Deferred fixes are forgotten |
| "One-time exception" | Exceptions become patterns |
| "Finish the task first" | Rule violations compound |

## Timing

Rule updates MUST happen BEFORE continuing the original task, IN the same session.

## Relationships

- **Verification**: [[r017]] — rule updates trigger re-verification
- **Pattern detection**: [[sys-naggy]] monitors 3+ recurring violations
- **Commit**: [[mgr-gitnerd]] for committing rule updates
