# R020: Completion Verification Rules

**Priority**: MUST | Before declaring [Done], verify actual completion against task-type-specific criteria. "Command executed" ≠ "succeeded."

## Overview

R020 prevents false completion declarations — the pattern of marking work done based on action taken rather than outcome achieved. Each task type has a specific verification matrix, and the agent must check actual outcomes (exit codes, file content, CI status) before declaring success.

## Task-Type Completion Matrix

| Task Type | Required Verification |
|-----------|----------------------|
| Release | All issues closed, version bumped, PR merged, GitHub Release created |
| Implementation | Compiles, tests pass, no TODO markers left |
| Documentation | Links valid, counts accurate, cross-refs updated |
| Git Operations | Exit code checked, working tree clean |
| Code Review | All findings addressed or deferred with justification |
| Agent/Skill Creation | Frontmatter valid, referenced skills exist, routing updated |

## Common False Completion Patterns

| Pattern | Reality | Fix |
|---------|---------|-----|
| "Command executed" | Exit code not checked | Check `$?` or tool output |
| "File created" | Content not verified | Read file back |
| "PR created" | CI not checked | Wait for CI green |
| "Tests pass" | Only subset run | Run full suite |

## Completion Contract Format

```
[Contract] Task: {name}
├── Criterion 1: {specific, verifiable condition}
└── Criterion N: {specific, verifiable condition}

[Done] Task: {name}
├── ✓ Criterion 1: {evidence}
└── ✓ Criterion N: {evidence}
```
