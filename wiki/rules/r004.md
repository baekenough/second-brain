# R004: Error Handling Rules

**Priority**: SHOULD | Error classification (Warning/Error/Critical), recovery strategies, and validation requirements.

## Overview

R004 defines a three-level error taxonomy and corresponding recovery strategies. The key distinction: Warning is informational (task continues), Error stops the current task (others can proceed), Critical stops everything and preserves state. This prevents cascading failures from being treated as warnings.

## Error Classification

| Level | Symptom | Response |
|-------|---------|----------|
| Warning | Task completes but needs attention | Output warning, continue |
| Error | Current task fails | Stop task, report cause, suggest alternative |
| Critical | Cannot proceed at all | Stop all, preserve state, report immediately |

## Error Report Format

```
[Error] {type} — Location: {file:line} — Cause: {cause} — Impact: {effect}
Attempted: 1. {try1} -> Failed  2. {try2} -> Failed
Recommended: {action1}, {action2}
```

## Recovery Strategies

| Type | Strategy |
|------|----------|
| Retryable | Retry up to 3x with backoff (1s, 2s, 4s), then report |
| Non-recoverable | Save state, rollback if possible, detailed report, wait for user |

## Validation Timing

- **Before action**: target exists, permissions available, dependencies met
- **After action**: expected = actual, file integrity, no unintended side effects
