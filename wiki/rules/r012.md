# R012: HUD Statusline Rules

**Priority**: SHOULD | Two-system status display: HUD events via stderr hooks, statusline via stdout command.

## Overview

R012 defines a two-channel status display architecture. HUD events (hook-based, stderr) appear inline in the conversation log for subagent spawn notifications. The statusline (command-based, stdout) provides persistent session status at the bottom of the screen.

## HUD Events (Hook-based)

```
─── [Spawn] {subagent_type}:{model} | {description} ───
```

- Implemented via `.claude/hooks/hooks.json` (PreToolUse → Agent/Task matcher)
- Used for multi-step, parallel, or long-running operations only

## Statusline (Command-based)

```
{Cost} | {project} | {branch} | RL:{rate_limit}% {countdown} | WL:{weekly_limit}% {countdown} | CTX:{usage}%
```

Config in `.claude/settings.local.json`:
- `statusLine.type: "command"`
- `statusLine.command: ".claude/statusline.sh"`
- Requires CC v2.1.80+ for RL/WL segments
- `refreshInterval` (v2.1.97+): auto-refresh interval in seconds

## Integration

| Rule | Interaction |
|------|-------------|
| R007 | Agent ID display correlates with HUD spawn events |
| R008 | Tool ID prefix appears before spawn HUD events |
| R009 | Parallel spawns show multiple HUD events simultaneously |

## Relationships

- **Agent ID**: [[r007]] — identification header complements HUD events
- **Tool ID**: [[r008]] — spawn announcements before HUD events
- **Parallel**: [[r009]] — parallel spawns generate multiple HUD notifications
