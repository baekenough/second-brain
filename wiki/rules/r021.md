# R021: Enforcement Policy

**Priority**: MUST | Advisory-first enforcement model — most rules use prompt engineering, with hard blocks reserved for critical safety boundaries.

## Overview

R021 documents the deliberate enforcement architecture: advisory-first, not hard-blocking. Hard blocks can trap agents in unrecoverable states; PostCompact re-injection prevents compaction amnesia. The goal is behavioral compliance through context, not mechanical blocking.

## Enforcement Tiers

| Tier | Mechanism | Rules | Behavior |
|------|-----------|-------|----------|
| Hard Block | PreToolUse hook, exit 2 | stage-blocker, dev-server tmux, rule-deletion-guard | Prevents tool execution |
| Soft Block | Stop hook prompt | R011 session-end saves | Auto-performs then approves |
| Advisory | PostToolUse hooks | R007, R008, R009, R010, R018 | Warns via stderr, never blocks |
| Prompt-based | CLAUDE.md + rules/ + PostCompact | All MUST rules | Behavioral guidance |

## Promoted to Hard Block

| Hook | Date | Justification |
|------|------|---------------|
| `rule-deletion-guard.sh` | 2026-04-08 | Rule files require individual confirmation before deletion |

## Hard Enforcement Candidates

| Rule | Candidate Hook | Promotion Condition |
|------|---------------|---------------------|
| R010 | git-delegation-guard.sh | >3 direct-write violations/session |
| R007/R008 | (new hook) | Identification omission rate >20% |

Promotion requires: measured violation data + user approval + rollback plan.

## Why Advisory-First

1. Agent flexibility: hard blocks can cause unrecoverable states
2. Graceful degradation: missing dependencies don't break sessions
3. Composability: external skills and internal rules coexist
4. PostCompact reinforcement: R007-R010/R018 re-injected after compaction
