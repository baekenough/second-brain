# R018: Agent Teams Rules

**Priority**: MUST (Conditional) | When `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` is set, Agent Teams is MANDATORY for qualifying collaborative tasks.

## Overview

R018 governs when to use Agent Teams (peer-to-peer mailbox coordination) versus the Agent tool (hierarchical subagent). When Agent Teams is enabled, it is not optional for qualifying tasks — it is required. The key differentiator is communication: subagents report only to the orchestrator, while Team members communicate peer-to-peer and share a task list.

## Decision Matrix

| Scenario | Preferred | Reason |
|----------|-----------|--------|
| Simple independent subtasks | Agent Tool | Lower cost, no overhead |
| Sequential-dependency init | Agent Tool | Blocked agents waste tokens |
| Research requiring discussion | Agent Teams | Iterative discovery |
| Code review + fix cycle | Agent Teams | Review → fix → re-review loop |
| 3+ agents with shared state | Agent Teams | Shared task list, peer messaging |
| Multi-issue release batch | Agent Teams | Coordinated release |
| Single file operations | Agent Tool | Overkill |

**Quick rule**: 3+ agents OR review cycle → Agent Teams. Sequential deps / scaffolding → Agent Tool.

## Lifecycle

```
TeamCreate → TaskCreate → Agent(spawn members) → SendMessage(coordinate)
  → TaskUpdate(progress) → ... → shutdown members → TeamDelete
```

## Spawn Completeness

All members must be spawned in a single message. Partial spawning is a violation of R018 and R009.

## Fallback

When Agent Teams unavailable: use Agent tool with R009/R010 rules.

## Relationships

- **Parallel**: [[r009]] — parallel execution gate for 2+ agents
- **Delegation**: [[r010]] — Teams members can spawn sub-agents as local orchestrators
- **Intent**: [[r015]] — transparency applies to Team creation decisions
